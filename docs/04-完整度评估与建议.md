## 二、技术债务清单

### 高优先级（影响功能可用性）

| 问题           | 影响               | 解决建议             |
| -------------- | ------------------ | -------------------- |
| 代码节点未实现 | 无法执行自定义逻辑 | 引入 vm2 沙箱执行 JS |
| 认证系统简陋   | 无真实用户管理     | 实现 JWT + 用户表    |
| 无 HTTP 节点   | 无法调用外部 API   | 添加 HTTP 请求节点   |
| 无沙箱机制     | 代码执行有安全风险 | 使用 docker 隔离     |

### 中优先级（影响用户体验）

| 问题         | 影响               | 解决建议         |
| ------------ | ------------------ | ---------------- |
| 无 Rerank    | 检索质量待提升     | 接入重排序模型   |
| 无图片输入   | 多模态能力不足     | 添加图片理解支持 |
| 无版本管理   | 工作流改错无法回滚 | 添加版本历史     |
| Redis 未启用 | 缓存未实际生效     | 修复 Redis 配置  |

### 低优先级（锦上添花）

| 问题       | 影响             | 解决建议            |
| ---------- | ---------------- | ------------------- |
| 无测试覆盖 | 代码质量难保障   | 补充单元测试        |
| 缺少 CI/CD | 部署流程未自动化 | 完善 GitHub Actions |
| 无文档导出 | 知识库无法分享   | 添加导出功能        |

## 四、关键决策建议

### 4.1 代码执行安全方案选型

| 方案        | 安全性     | 复杂度 | 性能 | 推荐度              |
| ----------- | ---------- | ------ | ---- | ------------------- |
| vm2         | ⭐⭐⭐     | 低     | 高   | ✅ 推荐（快速实现） |
| isolated-vm | ⭐⭐⭐⭐   | 中     | 高   | ✅ 推荐（生产环境） |
| Docker 沙箱 | ⭐⭐⭐⭐⭐ | 高     | 中   | 🔴 太重             |
| WebAssembly | ⭐⭐⭐⭐   | 高     | 高   | 🟡 学习成本大       |

**建议**：先用 `vm2` 快速实现，后期迁移到 `isolated-vm`

### 4.2 重排序模型选型

| 方案               | 准确率     | 成本         | 延迟 | 推荐度              |
| ------------------ | ---------- | ------------ | ---- | ------------------- |
| Cohere Rerank API  | ⭐⭐⭐⭐⭐ | 按调用付费   | 低   | ✅ 推荐（快速接入） |
| bge-reranker-v2-m3 | ⭐⭐⭐⭐   | 免费（本地） | 中   | ✅ 推荐（长期方案） |
| 无重排序           | ⭐⭐⭐     | 0            | 无   | ❌ 不推荐           |

**建议**：先接入 Cohere API 验证效果，再考虑本地部署

### 4.3 是否要实现多 Agent

| 方案          | 复杂度 | 价值          | 推荐度      |
| ------------- | ------ | ------------- | ----------- |
| 单 Agent      | 低     | 满足 80% 需求 | ✅ 当前足够 |
| 多 Agent 编排 | 高     | 复杂场景必需  | 🟡 二期考虑 |

**建议**：当前阶段专注单 Agent 体验，多 Agent 作为远期规划

## 六、给你的具体建议

### 如果你是技术小白

1. **先不要急着加功能**
   - 先通读文档，理解现有代码
   - 尝试运行和修改小功能

2. **从简单任务开始**
   - 修改 UI 样式
   - 添加一个新的节点类型（复制现有节点改改）

3. **多查资料**
   - 遇到不懂的概念（如向量、SSE）先搜索学习
   - 参考 Dify 的实现方式

### 如果你是开发者

1. **优先补代码节点**
   - 这是最大的功能缺口
   - 参考 n8n / Dify 的实现

2. **关注安全性**
   - 代码执行必须用沙箱
   - 用户输入要做校验

3. **考虑测试覆盖**
   - 工作流引擎是核心，要有单元测试
   - RAG 检索要准备测试数据集

### 如果你要商业化

1. **先验证需求**
   - 不要一次性做全功能
   - 找 3-5 个种子用户试用

2. **差异化定位**
   - 不要直接对标 Dify
   - 专注某一细分场景（如客服、知识问答）

3. **关注合规**
   - 用户数据隐私保护
   - AI 生成内容的免责声明
