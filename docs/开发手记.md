# AgentFlow Studio 开发手记

> 一个经济学系女生，是怎么在不会写代码的情况下，做出一个 AI 工作流平台的？

---

## 🌧 开始之前：一个彻底懵的文科生

我的专业是经济学。我的日常是看论文、跑 SPSS 回归、写分析报告。

在我的世界里，"后端"是指某家餐厅的后厨，"向量"是高中物理里有方向的那个箭头，"数据库"……我甚至认为 Excel 就是数据库。

然后有一天，我决定做一个 AI 工作流平台。

我也不知道为什么，可能是看了太多"这个是 Coze 的竞品"、"自己手搓一个 Agent 引擎"之类的帖子，脑子一热就开了个文件夹，起名叫 **AgentFlow Studio**。

那天晚上，我打开了人生中第一个 NestJS 项目，然后对着满屏的 `decorator`（装饰器？化妆品？）和 `dependency injection`（依赖注入？注射什么？），蒙了大概半个小时。

---

## 💡 第一周：开始理解"我到底在做什么"

我研究了一周市面上的 AI 平台，包括字节的 Coze、百度的文心一言工作流、Dify……

用经济学的眼光来看，这些产品的本质就是**资源调度**。用户的一个问题（输入）经过一系列"生产环节"（节点），最终变成一个答案（输出）。这不就是个生产函数嘛！

这个思路打通了我最初的认知壁垒。

于是我给自己的项目定了一个最简单的结构：

```
用户问一个问题
     |
     ↓
先去知识库翻翻（RAG 检索）
     |
     ↓
把找到的资料塞给 AI（LLM 节点）
     |
     ↓
AI 整理成漂亮的回答输出给用户
```

这三步，就是整个项目的灵魂。我把它画在了草稿纸上，贴在桌子旁边，每次看代码看不懂的时候就瞄一眼，提醒自己这一堆乱码是用来干啥的。

---

## 🔨 第二周：拖拽画布，第一次看到自己做的东西"动起来"

前端用的是 **Vue 3 + Vue Flow**。

Vue Flow 是一个拖拽连线库。简单来说，就是允许用户在屏幕上拖拽"节点"，然后用箭头把它们连起来，就像思维导图一样。

我设计了 7 种节点，每一种都有它的职责：

| 节点          | 我的理解（当时）     |
| ------------- | -------------------- |
| 触发节点 ▶️   | 工作流的"上班打卡机" |
| LLM 节点 🤖   | 真正负责思考的大脑   |
| 知识库节点 📚 | 翻资料的图书管理员   |
| 条件节点 ⚡   | 做选择的十字路口     |
| 代码节点 💻   | 处理复杂计算的小工具 |
| HTTP 节点 🌐  | 去外面拿数据的快递员 |
| 结束节点 ⏹️   | 告诉工作流"下班了"   |

有一个晚上，我把这些节点全都拖进了画布，然后用连线把它们串起来，点了一下"运行"按钮……

工作流执行了。

日志面板里滚动出了每个节点的状态。我盯着那个屏幕看了大概三秒钟，然后在出租屋里轻声说了一句"跑起来了"。

没有庆祝，只有一种说不清楚的踏实感。

## 📚 第三周：知识库——让 AI 真的"懂"我的文档

这一周处理的是最难啃的部分：**RAG（检索增强生成）**。

RAG 是什么？我的外行解释：就是你在问 AI 问题之前，先让它"看一遍"你的所有资料，然后它回答你的时候，是真的基于你的资料，而不是凭空编造。

整个流程分为三步：

### 第一步：文档切片（Chunking）

用户上传一个 PDF 或 TXT 文件，系统不能整篇吞下去，需要把它切成一小段一小段的——就像把一本书撕成一页一页，方便查找。

我用了一个叫 `RecursiveCharacterTextSplitter` 的工具，可以设置每段多少字、段与段之间重叠多少字（这样内容不会在切的地方断得太生硬）。

### 第二步：向量化（Embedding）

每一个文字段落，都会被转换成一串很长很长的数字（向量）。这串数字代表了这段话的"语义特征"，在数学上相似的概念，对应的向量也会更接近。

这是整个项目里我最花时间理解的概念。我当时的想法：**向量是在给文字"称体重"——相似的文字，重量会很接近。**

系统用的是 `pgvector` 扩展，把这些向量存进了 PostgreSQL 数据库。

### 第三步：混合检索（Hybrid Search）

用户提问时，系统同时使用两种方式找答案：

- **关键词搜索**（BM25）：字面上匹配，找含有相同词语的段落
- **向量搜索**：语义上匹配，找意思相近的段落

两者结合，再通过**重排序（Rerank）**综合打分，找出最相关的几段，塞进 AI 的上下文里。

当我第一次看到 AI 的回答里出现"（来源：知识库 #doc-001）"这个标注时，我差点把手机摔了（激动的那种）。它真的学会"引用文献"了。

## 💬 第四周：让对话有了灵魂——流式输出

最后一个核心功能是**对话界面**。

普通的接口是：你发一个请求，等几秒钟，AI 把整段答案一股脑返回给你。用户体验会很差，因为你不知道 AI 在干什么，只能干等。

我用的是 **SSE（Server-Sent Events）** 技术实现流式输出——让 AI 每生成一个字，就立刻推送给前端显示。就是你在 ChatGPT 里看到的那种"打字机"效果。

```
后端每生成一个 token → 立刻推给前端 → 前端实时拼接显示
```

这部分调试让我崩溃了两天。前端代码写完之后，死活收不到后端的流式数据。我把 Chrome 开发者工具的 Network 面板里每一条请求都研究了一遍，最后发现是响应头里少设置了一个 `Content-Type: text/event-stream`。

两天，一行配置。

我觉得这可能是整个项目里最有成就感的 bug 修复，因为修好之后，我第一次看到文字像水流一样出现在屏幕上，真的有一种"活了"的感觉。

## 🛠️ 最难的不是代码，是看懂错误信息

说实话，整个开发过程中，我真正自己从零"写出来"的代码极少。

我更多的时候是：

1. 描述我想要实现什么功能
2. 让 AI（Claude/Cursor）帮我生成代码
3. 运行，报错
4. 把报错信息贴给 AI，让它帮我分析
5. 修改，再运行
6. 重复 3-5，直到它跑通

但这个过程并不是简单的"复制粘贴"。因为如果你不理解每一步在做什么，你就没办法判断 AI 的答案是否正确，也没办法提出下一个有效的问题。

我的经济学背景反而在这里帮了很大的忙——我对**系统和流程**的理解，让我很容易画出每个模块的边界，判断一段代码应该放在哪里、它的输入输出是什么。

代码是 AI 帮我写的，但整个系统的**逻辑结构**，是我设计的。

## 📦 最终的项目长什么样

一键启动：

```bash
docker-compose up -d
```

然后打开浏览器，你会看到：

- **工作流编排页**：拖拽节点，连线，保存，运行，查看每个节点的执行日志
- **知识库页**：上传文件，自动分块，测试检索效果
- **对话页**：和你构建的智能体聊天，回答有来源溯源，打字机效果实时输出
- **监控页**：查看执行次数、检索时间、缓存命中率

整个项目包含：

- 前端：Vue 3 + Pinia + Vue Flow + Element Plus
- 后端：NestJS + TypeORM + PostgreSQL（含 pgvector 向量扩展）+ Redis
- AI 接入：支持 OpenAI 及所有兼容接口（包括 DeepSeek）

## 🌙 结尾：我学到了什么

学经济学让我习惯了思考"系统"——谁给谁输入，谁从谁取输出，中间经过了什么处理。这套思维方式，在构建一个软件系统时，比我想象中更加有用。

我没有成为一个"程序员"，我成为的是一个**可以用 AI 辅助工具把想法变成现实的人**。

区别在于：我不需要记住所有语法，但我需要清楚地知道我想要什么，以及为什么它现在还不是我想要的样子。

如果你也是文科生、商科生，也想做点什么技术相关的事情，我的建议是：

> **不要从"学编程"开始，要从"我要解决什么问题"开始。**

然后让 AI 陪你一起搞定它。
